# 文章发布助手

本技能用于将用户提供的文章源文件整理成符合本博客标准的文章，并放置到正确位置。

## 前置要求

在开始前，先确认一下当前工作目录是否在博客根目录（`D:\WorkSpace\blog`）。如果不在，先 cd 过去。

## 工作流程

### 第一步：读取源文件

用户会提供一个路径。这个路径可能是：

- **一个目录**：包含 `.md` 文件和图片（如 `~/Desktop/my-new-article/`）
- **一个 `.md` 文件**：单独的文章文件

使用 `Glob` 工具读取该路径下的所有文件。注意 `.md` 文件和图片文件（`.png`, `.jpg`, `.jpeg`, `.gif`, `.webp`, `.svg`）。

### 第二步：解析并校验 front matter

用 `Read` 读取 `.md` 文件的开头（front matter 区），检查以下字段：

| 字段 | 规则 | 缺省时处理 |
|------|------|-----------|
| `title` | 简体中文，文章标题 | 告诉用户需要提供 |
| `date` | `YYYY-MM-DDThh:mm:ss+08:00` | 使用当前时间 `2026-07-10T...` |
| `slug` | 全小写英文+连字符，用作 URL 标识 | 从标题自动生成（英文音译或拼音） |
| `lastmod` | 同 `date` 格式 | 同 `date` |
| `category` | 单个字符串（如 "随笔"） | 设默认值 `"随笔"` |
| `tags` | YAML 数组 | 空数组 `[]` |
| `description` | 一行摘要 | 从文章前几行提取 |
| `author` | 固定为 `Tianchi` | 补充为 `Tianchi` |

**为什么 slug 很重要**：slug 决定文章的永久链接 (`/:slug/`) 和目录名。如果用户没提供 slug，需要从 title 生成一个合理的英文标识。如果文章内容决定它适合放在某个特定 category（如"技术"、"读书"、"随笔"），请基于内容判断而非一律用默认值。

**date 与 lastmod 的关系**：如果文章是新文章，`lastmod` 等于 `date`。如果用户说这是修改已有的文章，保留原 `date`，更新 `lastmod` 为当前时间。

**校验完成后，向用户展示你打算填写的完整 front matter**，并确认是否要进行下一步。这个步骤让用户有机会修正你觉得不确定的字段（特别是 `title`、`slug`、`category`）。

### 第三步：确定目标位置

目标位置：`_posts/<YYYY-MM-DD-slug>/<YYYY-MM-DD-slug>.md`

- `YYYY-MM-DD`：从 `date` 字段提取日期部分
- `slug`：front matter 中的 `slug` 字段

如果文章已经在 `_posts/<YYYY-MM-DD-slug>/` 目录下且命名规范：

- 说明文章已在正确位置，只帮你提交（如果要求提交的话）

如果文章在别处（桌面、`_drafts/`、临时目录等）：

- 创建 `_posts/<YYYY-MM-DD-slug>/` 目录
- 将 `.md` 文件复制过去并重命名为 `<YYYY-MM-DD-slug>.md`
- 将所有图片文件复制到同一目录

**⚠️ 重要：不要删除源文件，使用复制操作。** 用户确认后再执行文件操作。可以在这一步询问用户是否还有其他需要补充的内容。

### 第四步：显示成果

整理完成后，用 Read 读取目标 `.md` 文件，向前确认一切正常。然后告诉用户：

- 文章路径
- 文章的 front matter 概况
- 如果有图片，列出图片文件

**除非用户明确要求，否则不要执行 `git commit` 或 `git push`。** 整理好放在那里，让用户自己决定提交时机。

### 第五步（可选）：提交

如果用户要求提交，执行：

```bash
git add _posts/<YYYY-MM-DD-slug>/
git commit -m "feat(post): <title>"
```

不 push。

## 关键原则

1. **先读后写**——永远先读源文件的内容，理解文章后再做判断，不要凭空臆测。
2. **确认再行动**——每次改变文件结构或内容前，先向用户展示你的计划并获得确认。特别是一定要展示你补全的 front matter 让用户过目。
3. **不破坏原文件**——使用复制而非移动。源文件留在原地。
4. **字段优先**——`slug`、`date`、`category` 这仨字段决定了文章的组织方式，尤其要确认。如果 slug 不会写，问问用户有没有英文标题或想要的 URL 后缀。
5. **理解文章内容再做判断**：category 应该从文章实际内容推断（技术类 vs 随笔 vs 读后感 vs 诗歌），不要一律设成"随笔"。读文章的前几段来判断它属于什么类型。
6. **中文博客属性**：博客是中文的，`title`、`description`、`category` 这些字段都用中文。只有 `slug` 和 `tags` 可能用英文。
